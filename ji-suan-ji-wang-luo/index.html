Chapter 1 计算机网络和因特网

网络核心

1. 分组交换

源端系统将要发送到目的端系统的长报文划分成较小的数据块，称为分组；

一条链路传输分组的时间 = 分组的比特长度L / 链路的传输速率R(bit/s)；

存储转发传输

交换机能开始像输出链路传输该分组的第一个比特之前，必须完整地接收整个分组；

通过由N条速率均为R的链路组成的路径，在源端系统和目的端系统之间有N-1个分组交换机，发送一个分组时，端到端的时延是
$$
d端到端=N\frac{L}{R}
$$

排队时延和分组丢失

如果一个路由器正在转发其它分组，那么新到达的分组就要在该路由器的输出缓存（输出队列）中排队等待，如果缓存区满，那么新到达的分组或者已经在排队的分组之一将被丢弃；

转发表和路由选择协议

每台路由器都有一个转发表，将目的地址映射为输出链路，当某一个分组到达路由器，路由器就会检查该分组的目的地址，搜索转发表，发现适当的输出链路；

2. 电路交换

电路交换网络中的复用

频分复用FDM：在连接期间每条连接专用一个频段

时分复用TDM：时间被换分为固定期间的帧，每个帧被划分为固定数量的时隙，网络为一条连接在每个帧中指定一个时隙；TDM一条电路的传输速率等于帧速率乘以一个时隙中的比特数量

FDM每条电路连续地得到部分带宽，TDM每条电路在时隙中周期性得到所有带宽；

分组交换网中的时延、丢包和吞吐量

1. 分组交换网中时延概述

节点处理时延、排队时延、传输时延、传播时延，这些时延之和是节点总时延

时延的类型

  • 节点处理时延：检查分组首部、处理分组导向何处
  • 排队时延：当分组在链路上等待传输时，会产生排队时延
  • 传输时延：将分组的所有比特推向链路所需的时间，L/R，（其中L是分组的比特长度，R是链路传输速率）
  • 传播时延：从链路的起点到目的端系统所需要的时间，以链路的传播速率传播，链路传播速率取决于链路的物理媒体，d/s，（其中d是两台端系统之间的距离，s是链路的传播速率）

2.排队时延和丢包

流量强度

假设用a表示分组到达队列的平均速率（pkt/s)，L表示分组长度，R表示链路传输速率，那么流量强度定义为：
$$
\frac{La}{R}
$$

当流量强度La/R > 1时，显然比特到达队列的平均速率超过了比特的传输速率，排队时延将趋于无穷大（不考虑丢包的情况下）；

丢包

当分组到达一个满的队列，由于没有地方储存该分组，路由器将其丢弃，该分组丢失；

3. 计算机网络中的吞吐量

假设主机A和主机B之间传输文件，则：

瞬时吞吐量：主机B（接收方）接收该文件的速率（bps）

平均吞吐量：文件比特/B接受所用的时间（F bits/ T s)

因特网协议栈的5个层次：物理层、链路层、网络层、运输层、应用层

  • 应用层：网络应用程序以及它们的应用层协议存留的地方，一个端系统中的应用程序使用应用层协议和另一个端系统的应用程序交换信息分组（报文）；

  • 网络层：将数据报的网络层分组从一台主机移动到另一台主机

Chapter 2 应用层

2.1. 应用层协议原理

2.1.1. 网络应用程序体系结构

客户机/服务器体系结构、P2P体系结构、客户机/服务器和P2P混合的体系结构

  • 客户机/服务器体系结构：
  • P2P体系结构：
    • 没有总是打开的服务器
    • 任意两台主机之间可以相互通信
    • 对等方向其他对等方提供服务，同时接受其他对等方的服务

2.1.2. 进程通信

进程：可以理解为运行在端系统上的应用程序；

同一主机上的两个进程通过内部进程通信机制进行通信；

不同主机上的进程通过交换报文相互通信

1. 客户机进程和服务器进程

客户机进程：发起通信的进程

服务器进程：等待联系的进程

2. 进程与计算机网络之间的接口

进程通过套接字向网络发送报文和接受报文；

套接字是 应用程序进程和运输层协议之间的接口；

3. 进程寻址

一台主机上运行的进程向另一台主机上运行的进程发送分组时，接收进程需要有一个地址，标识该接收进程地址需要知道：主机的IP地址和在目的主机中接收指定进程的标识符（端口号）；（主机虽然有唯一的32位IP地址，但是一台主机可以运行多个进程，所以只有主机的IP地址不能标识进程）

常用应用程序的端口号：

Web服务器：80

邮件服务器：25

FTP服务器：21

2.1.3. 因特网提供的运输服务

1. TCP服务

面向连接服务：在客户机程序和服务器程序之间必须建立连接；

可靠的数据传输服务：通信进程能够依靠TCP无差错，按适当顺序交付所有发送的数据；

流量控制：发送方不会淹没接收方；

拥塞控制：网络出现拥塞时抑制发送进程；

TCP没有提供时延保证、最小带宽保证

2. UDP服务

不可靠数据传输

没有提供：建立连接，可靠性，流量控制，拥塞控制，时延和带宽保证

2.2. Web和HTTP

2.2.1. HTTP概述

Web的应用层协议是超文本传输协议（HTTP）；

HTTP使用TCP作为支撑运输协议；

HTTP是无状态协议，不保存关于客户的任何信息；

2.2.2. 非持续连接和持续连接

1. 非持续连接

往返时间RTT：指一个短分组从客户机到服务器然后再往返客户机所花费的时间

$$ T_{总响应时间}=2RTT+t_{文件传输} $$

2. 持续连接

不带流水线的持久HTTP连接：客户只在收到前一个请求的响应后才发出新的请求，web页面所引用的每个对象都经历1个RTT的延迟，用于请求和接收该对象；

带流水线的持久HTTP连接：HTTP客户机每碰到一个引用就立即发出一个请求，因而HTTP客户可以一个接一个紧挨着发出各个引用对象的请求。服务器收到这些请求后，也可以一个接一个紧挨着发出各个对象。如果所有的请求和响应都是紧挨着发送的，那么所有引用到的对象一共只经历1个RTT的延迟

2.2.3. HTTP报文格式

HTTP报文分为请求报文和响应报文；

1. HTTP请求报文

HTTP请求报文的第一行是请求行，包括方法字段、URL字段和HTTP版本字段；

方法字段有：GET（绝大部分）、POST、HEAD、PUT、DELETE

$$ HTTP请求报文通用格式 $$

2. HTTP响应报文

HTTP响应报文的第一行是状态行，包括协议版本字段、状态码、相应状态信息；

Date：服务器产生并发送该响应报文的日期和时间；

Last-Modified：对象创建或者最后修改的日期和时间；

2.2.4. 用户与服务器的交互：cookie

cookie允许站点对用户进行跟踪

2.2.5. Web缓存

Web缓存器（代理服务器）能够代表初始Web服务器满足HTTP请求，可以大大减少客户请求的响应时间；

  • 客户机建立一个到Web缓存器的TCP连接，发送HTTP请求；
  • Web缓存器进行检查，若本地储存了请求对象的副本，则向客户机通过响应报文返回该对象；
  • 如果Web缓存器没有该对象，则Web缓存器建立一个到起始服务器的TCP连接，发送HTTP请求；初始服务器收到HTTP请求后，通过响应报文返回该对象；
  • Web缓存器接收该对象，存储副本，通过响应报文返回该对象给客户机

2.3. 文件传输协议FTP

  • FTP客户首先发起建立1个与FTP服务器端口号21之间的TCP控制连接, 指定TCP作为传输层协议；
  • 客户在建立的控制连接上获得身份认证；
  • 客户在建立的控制连接上发送命令浏览远程主机的目录；
  • 当服务器接收到1个文件传输命令时，在服务器端口号20创建1个与客户的TCP数据连接；
  • 文件传输完成后，服务器结束这个TCP数据连接

2.4. 因特网中的电子邮件

电子邮件三个主要组成部分：用户代理、邮件服务器、简单邮件传输协议SMTP

2.4.1. SMTP

SMTP使用持久连接；

SMTP要求邮件消息采用7比特ASCII码表示；

SMTP服务器使用CRLF.CRLF来判断邮件消息的结束；

SMTP和HTTP的对比（区别）：

  • SMTP是推协议，发送邮件服务器把文件推向邮件接收服务器，TCP是由想发送文件的机器发起的；而HTTP是拉协议，TCP连接是由想要接收文件的机器发起的；
  • SMTP要求每个报文采用7比特ASCII码格式，HTTP数据不受这种限制；
  • HTTP把每个对象单独封装到它自己的HTTP响应报文中，而SMTP把所有报文对象放到一个报文中

2.4.2. 邮件报文格式

From: alice@crepes.fr//必须有
To: bob@hamburger.edu//必须有
Subject: Searching for the meaning of life.//可有，也可以没有

2.5. DNS 因特网的目录服务

因特网上的主机可以通过主机名和IP地址（4个字节、32位）进行标识；

2.5.2. DNS提供的服务

DNS协议运行在UDP之上，使用53端口号；

DNS（域名系统）的主要服务：提供主机名到IP地址的转换；

主机别名：一个主机可以有一个规范主机名和多个主机别名，（主机别名便于人们记忆），应用程序可以调用DNS来获取主机别名对应的规范主机名和主机的IP地址；

邮件服务器别名：同上；

负载分配：DNS实现冗余服务器：一个IP地址集合可以对应于同一个规范主机名；

2.5.2. DNS工作机理

假设只使用一个DNS服务器，即集中式设计，那么就会产生如下问题：

  • 单点故障：这一个DNS服务器崩溃，整个因特网就会瘫痪；
  • 通信容量：这个DNS服务器要处理所有的DNS查询，通讯量巨大；
  • 远距离集中式数据库:单个DNS服务器不可能临近所有的查询客户;
  • 维护困难：保存记录信息、频繁更新；
  • 无扩展能力

1. 分布式、层次数据库

为了解决扩展性问题、DNS由大量DNS服务器，以层次方式组织、分布在世界各地；

层次结构分为根DNS服务器、顶级DNS服务器、权威DNS服务器；

本地DNS服务器：起着代理的作用，转发到层次结构中；

DNS查询可分为递归查询和迭代查询，一般情况下，从请求主机到本地DNS服务器的查询是递归查询，其余的查询是迭代查询；

2. DNS缓存

一旦某个DNS服务器获得DNS映射, 它将缓存该映射到局部内存；

由于主机和主机名与IP地址间的映射不是永久的，服务器在一定时间后将丢弃缓存的信息；

本地DNS服务器可以缓存TLD服务器（顶级DNS服务器）的IP地址，因此根DNS服务器不会被经常访问；

3. DNS记录与报文

共同实现DNS分布式数据库的所有DNS服务器存储了资源记录（RR），RR提供主机名到IP地址的映射；

RR的格式：(Name, Value, Type, TTL)

TTL是该记录的生存时间，决定RR应当删除的时间；

Name 和 Value 的值取决于 Type:

  • Type = A, Name = 主机名, Value = IP地址;
  • Type = NS, Name = 域名, Value = 该域权威名字服务器的主机名;
  • Type = CNAME, Name = 主机别名, Value = 规范主机名;
  • Type = MX, Name = 邮件服务器的主机别名, Value = 邮件服务器的规范主机名;

Chapter 3 运输层

3.1. 运输层服务

运输层协议运行在端系统：

  • 发送端将从发送应用程序进程接收到的报文转换成运输层分组（报文段）；
  • 接收端将数据段重新组装成报文传递给应用层；

3.1.1. 因特网运输层概述

IP的服务模型是尽力而为交付服务，尽最大的努力在通信主机之间交付报文段，但是不确保报文段的交付、顺序和完整性，因此被称为不可靠服务；

UDP和TCP最基本的责任是将两个端系统之间IP的交付服务扩展为运行在端系统上的两个进程之间的交付服务（运输层的多路复用与多路分解）；

3.2. 多路复用与多路分解

在接收主机多路分解：将运输层报文段中的数据交付到正确的套接字；

在发送主机多路复用：从不同的套接字收集数据块，为每个数据块封装首部信息生成报文段，然后将报文段传递到网络层；

3.2.1. 无连接和面向连接的多路复用和多路分解

UDP套接字两个指定因素：目的IP地址、目的端口号；

  • 具有不同的源IP地址且/或源端口号，但具有相同的目的IP地址和目的端口号的IP数据报指向同样的套接字；

TCP套接字的四个指定因素：源IP地址、源端口号、目的IP地址、目的端口号；

  • 两个具有不同源IP地址或源端口号的报文段将被定向到两个不同的套接字；
  • 服务器主机支持很多并行的TCP套接字；

3.3. 无连接运输-UDP

3.3.1. UDP报文段结构

UDP首部有4个字段，每个字段有两个字节；

长度字段指示了UDP报文段中的字节数（包括首部和数据），以字节为单位；

接收方使用校验和检查报文段是否出现差错；

3.3.2. UDP校验和

发送发将数据段看作16比特的01序列，将所有的16比特字的和（遇到溢出则回卷）进行反码运算，得到的结果放到UDP报文段中的检验和字段；

接收方将全部的16比特字的和加上校验和，如果结果16位全为1，则没有检测到错误，反之则检测到错误；

3.4. 可靠数据传输原理

3.4.1. 构造可靠数据传输协议

1. 经完全可靠信道的可靠数据传输rdt1.0

假设信道完全可靠，不会出现bit错误和分组丢失；

2. 经具有bit错误信道的可靠数据传输rdt2.0（停等协议）

假设所有发送的分组按照发送的顺序被接收；

自动重传请求协议（Automatic Repeat reQuest， ARQ）：使用肯定确认和否定确认，使得发送发知道哪些分组被正确接收，哪些分组没被正确接收；

当发送方处于等待ACK或NAK状态时，发送发不能从上层获得更多数据，rdt_send()事件不可能出现，所以重传时发送方不会发送一块新数据，像rdt2.0这样的协议被称为停等协议；

由于ACK和NAK分组也可能出现差错，导致发送方无法知道接收方是否正确接收了上一个分组，所以还需要能够纠正ACK和NAK分组中出现的差错：

rdt2.1
$$ rdt2.1发送方（0和1交替使用） $$ $$ rdt2.1接收方 $$
rdt2.2

发送方收到重复ACK将重发当前分组；

3. 经具有比特差错的丢包信道的可靠数据传输rdt3.0（比特交替协议）

如果发送方在一个确定的时间内没有收到ACK，则重传当前分组，发送方在发送任何一个分组时，都要启动一个定时器；

3.4.2. 流水线可靠数据传输协议

rdt3.0是一个停等协议，因此发送发利用率较低，解决性能问题的一个简单方法是以不停等方式运行，允许发送发发送多个分组而无需等待确认；

流水线：

  • 需要增加序号的范围：每个输送中的分组都必须有唯一的序号，同时也可能有多个在输送中的未确认报文；
  • 发送发和接收方需要能够缓存一定数目的分组；
  • 解决流水线的差错恢复的两种基本方法：回退N步（Go-Back-N，GBN）、选择重传（SR）；

3.4.3. 回退N步-GBN（滑动窗口协议）

  • 发送方能够最多发送N个未被确认的分组；
  • 接收方仅发送累积ACK，假如收到分组与前面的分组之间有间隙，则不发送ACK；
  • 发放方为最早未被确认的分组（基序号）启动定时器，当定时器超时，重传所有未被确认的分组；
  • 累积确认：接收方总是为正确接收的最高序号的分组发送ACK；
  • 接收方丢弃所有失序分组，所以接收方只需要维护下一个按序接收分组的序号；

3.4.4. 选择性重传-SR

  • 发送方对每个没有确认的分组都启动一个定时器；

必须限定SR协议窗口大小，窗口太大可能区分不了一个分组是重传分组还是下一个新的分组；
$$
{\color{red}window}\space {\color{red}size \le \frac{sequnm}{2}}
$$

3.5. 面向连接的运输：TCP

3.5.1. TCP报文段结构

TCP报文段的首部一般是20字节（UDP报文段首部一般是8字节）；

源端口号、目的端口号：用于多路复用或多路分解来自于上层应用的数据；

1. 序号、确认号

序号：一个报文段的序号是该报文段首字节在字节流中的位置编号；

确认号：期望从另外一边收到的下一个字节的序号，（累积确认）；

3.5.2. 往返时间的估计与超时

1. 估计往返时间

定义变量：

SampleRTT:报文段的样本RTT，表示某报文段从发出到确认报文被接收的时间间隔；

EstimatedETT:SampleRTT均值；

DevRTT:RTT偏差；

TimeoutInterval:重传超时间隔；

则：
$$
EstimatedRTT=(1-\alpha)\cdot EstimatedRTT+\alpha\cdot SampleRTT
$$
通常α取0.125，所以上述公式可记为：
$$
EstimatedRTT=0.875\cdot EstimatedRTT+0.125\cdot SampleRTT
$$

2. 估算SampleRTT偏离EstimatedRTT的程度

$$
DevRTT = (1-\beta)\cdot DevRTT+\beta\cdot|SampleRTT-EstimatedRTT|
$$

通常β取0.25；

3. 设置重传时间间隔

$$
TimeoutInterval=EstimatedRTT+4\cdot DevRTT
$$

3.5.3. 可靠数据传输

1. ACK的产生

2. 快速重传

发送发在超时事件发生之前通过冗余ACK检测丢包情况，如果TCP发送方接收到对相同数据的3个冗余ACK，说明这个已经被确认三次的报文段之后的报文段已经丢失，就会执行快速重传，在该报文段的定时器过期之前重传丢失的报文段；

3.5.4. 流量控制（不同于拥塞控制）

TCP为它的应用程序提供了流量控制服务，消除发送方使接收方缓存溢出的可能性；

接收方在TCP头部消息的rwnd字段（接收窗口字段）通知发送方剩余缓冲区的大小，

  1. 对于接收方，定义变量：

LastByteRead:接收方的应用进程从缓存读出的数据流的最后一个字节的编号；

LastByteRcvd:从网络中到达并且已经放入接收方缓存中的数据流的最后一个字节的编号；

RcvBuffer:接收方接收缓存大小；

则：
$$
LastByteRcvd - LastByteRead\le RcvBuffer\space
$$

$$
rwnd =RcvBuffer-\left [ LastByteRcvd - LastByteRead\right ]
$$

  1. 对于发送方，定义变量：

LastByteSent:已经发送的数据流中的最后一个字节的编号；

LastByteACKed:已经确认的数据流中的最后一个字节的编号；

则：
$$
LastByteSent-LastByteACKed\le rwnd
$$

3.5.5. TCP连接管理

TCP报文段结构中，有几个特殊的字段：

  • 确认ACK：占1位，仅当ACK=1时，确认号字段才有效。ACK=0时，确认号无效；
  • 同步SYN：连接建立时用于同步序号。当SYN=1，ACK=0时表示：这是一个连接请求报文段。若同意连接，则在响应报文段中使得SYN=1，ACK=1。因此，SYN=1表示这是一个连接请求，或连接接受报文。SYN这个标志位只有在TCP建产连接时才会被置1，握手完成后SYN标志位被置0；
  • 终止FIN：用来释放一个连接。FIN=1表示：此报文段的发送方的数据已经发送完毕，并要求释放运输连接；

1. TCP建立连接的三次握手

  • 主机A向主机B发送TCP连接请求报文，其中包含主机A的初始序列号seq(A)=x。（其中报文中同步标志位SYN=1，ACK=0，表示这是一个TCP连接请求数据报文；序号seq=x，表明传输数据时的第一个数据字节的序号是x）；
  • 主机B收到请求后，会发回连接确认报文。（其中确认报文段中，标识位SYN=1，ACK=1，表示这是一个TCP连接响应数据报文，并含主机B的初始序列号seq(B)=y，以及主机B对主机A初始序列号的确认号ack(B)=seq(A)+1=x+1）；
  • 第三次，主机A收到主机B的确认报文后，还需作出确认，即发送一个序列号seq(A)=x+1；确认号为ack(A)=y+1的报文；

为什么需要第三次握手?
还要再发送一次确认是为了，防止已失效的连接请求报文段突然又传到了B，因而产生错误。
已失效的报文段：正常情况下：A发出连接请求，但因为丢失了，故而不能收到B的确认。于是A重新发出请求，然后收到确认，建立连接，数据传输完毕后，释放连接，A发了2个，一个丢掉，一个到达，没有“已失效的报文段”
但是，某种情况下，A的第一个在某个节点滞留了，延误到达，本来这是一个早已失效的报文段，但是在A发送第二个，并且得到B的回应，建立了连接以后，这个报文段竟然到达了，于是B就认为，A又发送了一个新的请求，于是发送确认报文段，同意建立连接，假若没有三次的握手，那么这个连接就建立起来了（有一个请求和一个回应），此时，A收到B的确认，但A知道自己并没有发送建立连接的请求，因为不会理睬B的这个确认，于是呢，A也不会发送任何数据，而B呢却以为新的连接建立了起来，一直等待A发送数据给自己，此时B的资源就被白白浪费了。但是采用三次握手的话，A就不发送确认，那么B由于收不到确认，也就知道并没有要求建立连接。
简而言之：第三次握手，主机A发送一次确认是为了防止：如果客户端迟迟没有收到服务器返回的确认报文，这时他会放弃连接，重新启动一条连接请求；但问题是：服务器不知客户端没收到，所以他会收到两个连接请求，白白浪费了一条连接开销。

2. TCP释放连接四次握手

  • 关闭客户端到服务器的连接：首先客户端A发送一个FIN，用来关闭客户到服务器的数据传送，然后等待服务器的确认。其中终止标志位FIN=1，序列号seq=u;

  • 服务器收到这个FIN，它发回一个ACK，确认号ack为收到的序号加1;

  • 关闭服务器到客户端的连接：也是发送一个FIN给客户端;

  • 客户段收到FIN后，并发回一个ACK报文确认，并将确认序号seq设置为收到序号加1;

  • 首先进行关闭的一方将执行主动关闭，而另一方执行被动关闭

3.6. 拥塞控制原理

拥塞控制方法：

  • 端到端拥塞控制（TCP所采用）：端系统没有从网络中得到明确的反馈，需要通过观察网络行为（分组丢失、时延）推断出拥塞；
  • 网络辅助的拥塞控制：路由器给发送方提供关于网络中拥塞状态的反馈信息（如用一个比特指明链路中的拥塞情况、路由器显示地通知发送方该路由器能在输出链路上支持的最大发送速率）

3.7. TCP拥塞控制

运行在发送方的TCP拥塞控制机制跟踪一个变量–拥塞窗口（cwnd），在一个发送方中，未被确认的数据量不会超过cwnd和rwnd的最小值；
$$
LastByteSent - LastByteACKed\le min{cwnd, rwnd}
$$
TCP拥塞控制算法

  • 慢启动
  • 拥塞避免
  • 快速恢复

  1. 慢启动

    在慢启动状态，cwnd的值以一个MSS开始，每当传输的报文段首次被确认就增加一个MSS（指数增长）；

    如果存在一个超时指示的丢包事件，TCP发送方将慢启动阈值ssthresh设置为当前cwnd的一半，然后将cwnd设置为1，重新开始慢启动过程；

    当cwnd的值大于ssthresh时，结束慢启动，TCP转移到拥塞避免模式；

  2. 拥塞避免

    每个RTT只将cwnd的值增加一个MSS；

  3. 快速恢复

4.3. 网际协议：IPv4、寻址、IPv6

4.3.1. IPv4数据报格式

  • IP协议版本号：这4比特规定了数据报的IP协议版本，路由器通过版本号确定如何解释IP数据报的剩余部分；
  • 首部长度：IPv4数据报可以包含一些可变数量的选项，但是大多数IP数据报不包含选项，所有一般IP数据报的首部为20字节；
  • 数据报长度：总长度包含首部和数据，长为16比特，所以IP数据报的理论最大长度为65535字节
  • 标识、标志、片偏移：与IP分片有关；
  • 寿命TTL：确保数据报不会永远在网络中循环，每当一台路由器处理数据报时，寿命字段的值减一，当TTL字段减为零时，该数据报必须被丢弃；

4.3.2. IPv4数据报分片

并不是所有链路层协议都能承载相同长度的网络层分组，一个链路层帧能承载的最大数据量叫做最大传输单元（MTU）；

当生成一个数据报时，发送主机为该数据报设置源地址和目的地址的同时贴上标识号，发送主机通常将它发送的每个数据报的标识号加1；

当某路由器需要对一个数据报分片时，形成的每个数据报（每个分片）具有初始的源地址、目的地址、标识号；

由于IP是一种不可靠的服务，一个或多个片可能永远也到达不了目的地，所以为了让目的主机相信它已经收到了初始数据报的最后一个片，最后一个片的标志比特设为0，所有其它分片的标志比特置1；

为了让目的主机知道是否丢失了一个分片，并且能够按照正确的顺序重新组装分片，使用偏移字段指定该片应该放在初始IP数据报的哪个位置；

4.3.3. IPv4编址

接口：主机与物理链路之间的边界，一般来说主机有一个接口，路由器有多个接口；

由于IP要求每个主机或路由器接口都要拥有自己的IP地址，所以一个IP地址与一个接口相关联；

每个IP地址的长度是32比特（4字节），所以共有$2^{32}$个可能的IP地址，这些地址通常采用点分十进制记法书写，即地址中的每个字节用它的十进制形式书写，各个字节之间用点隔开，例如193.32.216.9；

一个接口的IP地址需要由其连接的子网决定；
